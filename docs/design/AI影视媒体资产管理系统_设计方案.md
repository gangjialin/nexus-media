## 一、用户画像与场景

### 1.1 人物画像

#### 导演 · 张导（45岁）

> "我不关心技术细节，我只要打开电脑就知道各组在干什么、问题卡在哪了。"

| 维度 | 描述 |
|------|------|
| 角色 | 总导演，同时把控 2-3 个项目 |
| 技术能力 | 低 — 会用 iPad 看剧本、用微信发语音 |
| 常用设备 | 办公室 iMac、随身 iPad Pro |
| 核心痛点 | 不知道各组真实进度；审阅意见总有人漏看 |
| 使用习惯 | 每天早上一杯咖啡的时间打开系统看全局，然后开早会 |

**典型日**：打开系统 → 看全局看板（扫一眼各组成片进度）→ 打开待审阅队列 → 逐条看组长提上来的成片 → 边看边批注 → 一键驳回或通过

#### 组长 · 李姐（32岁）

> "我不要再来回传百度网盘了，一个 5G 的视频上传下载一上午没了。"

| 维度 | 描述 |
|------|------|
| 角色 | 动画组组长，带 6 人小组 |
| 技术能力 | 中 — 会用剪辑软件，熟悉基本办公工具 |
| 常用设备 | MacBook Pro + 双显示器 |
| 核心痛点 | 组员交上来的版本管理混乱；导演意见转达容易走样 |
| 使用习惯 | 每天多次登录，上传/审阅/分配是高频操作 |

**典型日**：收通知有新任务 → 分配给组员 → 组员提交了成品 → 在线审阅 → 标注意见 → 退回修改或提交导演 → 有空时整理本组素材资产

### 1.2 核心用户旅程

#### 场景一：剧本从定稿到分配（导演视角）

```
定稿剧本 → 按场次分配 → 各组收通知 → 组长再细分
```

**详细的步骤：**

1. 导演在"剧本管理"中打开已定稿的剧本
2. 系统按场次自动拆分，左侧显示场景列表
3. 导演选择场景"第 3 场 - 火车站离别"
4. 点击"分配"按钮 → 选择目标制作组（A 组/B 组/C 组）
5. 填写任务说明、截止日期
6. 提交 → 系统通知组长
7. 组长收到通知 → 打开任务 → 按镜头分配给组员

#### 场景二：素材上传 → 制作 → 提交 → 审阅 → 修改 → 定稿

这是系统最核心的业务闭环：

```
上传素材 → AI自动打标签(二期) → 制作完成 → 提交组长审阅
    → 组长批注 → 退回修改 or 通过 → 提交导演审阅
    → 导演批注 → 退回修改 or 定稿
```

#### 场景三：视频审阅全流程

1. 导演/组长在"待审阅"列表中点击一个视频
2. 播放器自动加载（先加载代理文件，秒开）
3. 边播放边在关键帧暂停
4. 点击"添加批注" → 暂停播放 → 可圈出画面区域 → 写文字 → @责任人
5. 批注自动汇总到右侧面板，按时间排序
6. 所有批注添加完毕后，点击"提交审阅意见"
7. 被 @ 的人收到通知，打开后逐条处理
8. 处理完一条标记"已修改"
9. 组长/导演确认后标记"通过"，流程闭环

## 三、核心功能流程

### 3.1 剧本管理全流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 导入剧本  │ → │ 场景拆分  │ → │ 在线编辑  │ → │ 版本定稿  │
│ PDF/DOCX │    │ 自动识别  │    │ 多人协同  │    │ 锁定版本  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                      │
                                                      ▼
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 批注闭环  │ ← │ 导演审阅  │ ← │ 任务执行  │ ← │ 按场次分配│
│ 跟踪确认  │    │ 划词批注  │    │ 各组制作  │    │ → 组长   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

**状态流转**：

```
待导入 → 编辑中 → 待审阅 → 修改中 → 已定稿
                                    → 已分发
                                    → 各组制作中
```

### 3.2 任务分配与执行流

```
┌──────────────────┐
│ 导演选择场次     │
│ + 指定制作组     │
└────────┬─────────┘
         ▼
┌──────────────────┐     ┌──────────────────┐
│ 组长收到通知     │  →  │ 组长按镜头       │
│ 查看分配内容     │     │ 细分给组员       │
└──────────────────┘     └────────┬─────────┘
                                  ▼
                         ┌──────────────────┐
                         │ 组员收到任务通知  │
                         │ "我要做什么"      │
                         └────────┬─────────┘
                                  ▼
                         ┌──────────────────┐
                         │ 组员下载素材      │
                         │ → 制作 → 上传    │
                         │ → 提交组长审阅   │
                         └────────┬─────────┘
                                  ▼
               ┌─────────────────────────────┐
               │ 组长审阅                     │
               │ ├─ 通过 → 提交导演审阅       │
               │ └─ 驳回 → 退给组员修改       │
               └─────────────────────────────┘
                                  ▼
               ┌─────────────────────────────┐
               │ 导演审阅                     │
               │ ├─ 通过 → 定稿 / 归档        │
               │ └─ 驳回 → 退给组长/组员修改  │
               └─────────────────────────────┘
```

**任务状态机**：

```
待分配 → 制作中 → 组长审阅
    ↑         ↓      ↓
    │     组长驳回  提交导演
    │         ↓      ↓
    └── 修改 ─┘   导演审阅
                      ↓
                   ├─ 定稿
                   └─ 导演驳回 → 修改 → 重新提交
```

### 3.3 视频审阅流程

```
┌─────────────────────────────────┐
│     审阅发起方（导演/组长）      │
│                                 │
│  1. 从待审阅列表打开视频        │
│  2. 暂停 → 添加时间点批注       │
│  3. 可选：在画面圈选区域        │
│  4. 输入文字 @责任人            │
│  5. 继续播放 → 重复 2-5         │
│  6. 点击"提交审阅意见"          │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│     被 @ 责任人（组员）          │
│                                 │
│  1. 收到通知                    │
│  2. 打开视频                    │
│  3. 查看批注列表                │
│  4. 点击每条批注 → 跳转到对应帧 │
│  5. 修改 → 标记"已修改"         │
│  6. 全部完成后→"申请重新审阅"    │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│     审阅发起方                   │
│                                 │
│  1. 收到"申请重新审阅"通知       │
│  2. 打开视频 → 逐条确认         │
│  3. 确认通过 → 标记"通过"       │
│    仍有问题 → 退回 → 循环       │
└─────────────────────────────────┘
```

**批注状态流转**：

```
发起方提交    责任人修改    发起方确认
─────────→  ─────────→  ─────────→
 待处理        已修改       已确认
  │                            │
  │ 发起方驳回                │
  └────────────────────────────┘
```

## 六、业务规则定义

### 6.1 剧本版本规则

```
版本号格式: v{数字}（如 v1、v2、v8）
规则:
- 每次手动保存或自动保存触发新版本？→ 手动"定稿"操作才创建版本节点
- 自动保存属于草稿，不创建版本
- 只有"已定稿"的版本才能被分发
- 版本对比：任意两个版本之间可以对比差异（高亮增减内容）
- 版本回退：回退操作会创建新版本，不覆盖历史
```

### 6.2 审阅状态流转规则

```
批注状态:
  待处理 → 修改中 → 已修改 → 已确认
    │         │                  │
    └── 仅发起方可直接驳回 ───────┘
    └── 责任人超时 3 天未处理 → 自动升级通知组长

视频整体状态（由批注汇总决定）:
  全部已确认 → 视频状态为"审阅通过"
  存在驳回 → 视频状态为"修改中"
  全部已修改待确认 → 视频状态为"待确认"
```

### 6.3 权限校验规则

```
- 用户只能看到自己角色范围内 + 被分配的项目/任务
- 组员不能看到本组其他人的任务详情
- 外部方不能看到除被分享内容外的任何信息
- 分享链接如设置了密码，试错 5 次后锁定 30 分钟
- 水印：包含访客姓名 + 手机号后四位 + 当前时间
```

### 6.4 文件规则

```
- 允许上传的最大单个文件: 100GB（可配置）
- 允许上传文件类型: 见需求文档 V2 4.2.1 节
- 文件名自动清理特殊字符，保留 Unicode
- 文件去重: 以 SHA256 哈希为准，同内容文件仅保留一份
- 临时文件保留 7 天后自动清理
- 回收站保留 30 天
```

### 6.5 外部链接规则

```
- 有效期最长 30 天（可配置）
- 未设置密码时为公开链接，设置密码后需要密码访问
- 水印为必选项（不可关闭）
- "仅在线"模式下禁止下载（浏览器层面 + 服务端双重校验）
- 链接可随时吊销
- 访问日志保留 180 天
```

## 九、API 设计

### 9.1 API 设计原则

- **RESTful** 风格，资源名使用复数
- 版本前缀 `/api/v1/`
- 统一响应格式
- 认证通过 Bearer Token（JWT）
- 分页统一使用 `?page=1&page_size=20`
- 排序统一使用 `?sort=-created_at`（负号表示降序）

**统一响应格式**：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 100
  }
}
```

**错误响应**：

```json
{
  "code": 40001,
  "message": "剧本版本不存在",
  "details": null
}
```

### 11.4 各 AI 功能实现方案

#### AI 分镜生成

```
输入: 剧本段落文字（含场景描述、对白）
流程:
  1. 构建 Prompt → 包含分镜格式模板
  2. 调用 LLM API (通义千问 / GPT-4o)
  3. 解析结构化输出（JSON）
  4. 返回分镜表

Prompt 结构:
  - System: 你是资深影视分镜师，根据以下剧本生成专业分镜表
  - User: [场景描述 + 对白]
  - Output format: 镜头号 | 景别 | 运镜 | 画面描述 | 对白 | 预估时长

示例输出:
  [
    {"shot": 1, "framing": "远景", "movement": "固定",
     "description": "火车站全景，候车厅人群熙攘",
     "dialogue": "", "duration": "5s"},
    {"shot": 2, "framing": "中景", "movement": "推",
     "description": "角色A拖着行李箱走入画面，神情焦虑",
     "dialogue": "人呢？不是说好了在这里等吗？", "duration": "8s"}
  ]
```

#### AI 概念图生成

```
输入: 场景描述文字（风格、氛围、构图参考）
流程:
  1. 构建正向 Prompt + 负向 Prompt（避免内容）
  2. 调用 Stable Diffusion API
  3. 图像后处理（分辨率调整、水印叠加）
  4. 返回图片URL

Prompt 结构:
  - 正向: "电影感，火车站候车大厅，黄昏光线，35mm镜头，电影色调"
  - 负向: "low quality, blurry, distorted, extra limbs"
  - 参数: steps=30, guidance=7.5, size=1024x576
```

#### AI 自动标签（二期）

```
输入: 视频文件 / 视频 URL
流程:
  1. 场景分割 → 提取关键帧（每秒1帧或场景切换时）
  2. 对每帧并行执行:
     ├─ CLIP: 场景分类（室内/室外/城市/自然...）
     ├─ YOLO: 物体检测（人/车/道具/动物...）
     ├─ FaceNet: 人脸识别（匹配演员库）
     └─ PaddleOCR: 文字提取（路牌/标题/字幕）
  3. ASR 转录 → 提取关键词和主题
  4. 聚合结果 → 生成标签（去重、按置信度排序）
  5. 写回数据库

标签格式:
  [
    {"tag": "火车站", "type": "scene", "confidence": 0.95,
     "timestamps": [[0, 300], [500, 800]]},
    {"tag": "角色A", "type": "face", "confidence": 0.92,
     "timestamps": [[100, 250]]},
    {"tag": "行李箱", "type": "object", "confidence": 0.88,
     "timestamps": [[0, 100]]}
  ]
```

#### AI 语义搜索（二期）

```
索引阶段:
  1. 对每个视频 → 场景分割 → 每段提取关键帧
  2. 每帧生成字幕/描述（BLIP2 或 LLM 描述）
  3. 描述文本 → 向量嵌入（text2vec / BGE）
  4. 向量存入 Milvus + 关联 asset_id + 时间戳

搜索阶段:
  1. 用户输入自然语言："一个穿红色衣服的人在雨中奔跑"
  2. 文本 → 同一模型转为向量
  3. Milvus 向量相似度搜索（Top-K）
  4. 返回匹配结果 [{asset_id, timestamp, similarity, thumbnail}]
  5. 用户点击 → 跳转到视频对应时间点

性能指标:
  - 搜索响应 < 2 秒（500 万向量时）
  - 召回率 > 85%（Top-10）
```

### 11.5 模型部署策略

| 模型 | 部署方式 | 硬件要求 | 预估成本 |
|------|---------|---------|---------|
| 通义千问 / GPT-4o | API 调用 | 无 | 按 Token 计费 |
| Stable Diffusion | GPU 云实例 / API | 16GB+ VRAM | 按生成次数 |
| Whisper | 本地 / GPU 云 | 8GB+ VRAM（large） | 低（本地部署） |
| YOLOv11 | 本地 GPU / CPU | 6GB+ VRAM | 低 |
| CLIP | 本地 GPU | 8GB+ VRAM | 低 |
| text2vec | 本地 CPU/GPU | 4GB+ RAM | 极低 |
| Milvus | 云实例 / 自托管 | 取决于数据量 | 中 |

> **一期建议**：LLM 和图像生成走 API，降低初期运维成本。YOLO/CLIP 等推理模型本地部署，避免传输视频文件的带宽成本。

## 十三、安全设计

### 13.1 认证与授权

| 层级 | 方案 | 说明 |
|------|------|------|
| **传输层** | HTTPS TLS 1.3 | 全站 HTTPS，HSTS 头 |
| **认证** | JWT（access + refresh token） | access 15 分钟过期，refresh 7 天 |
| **RBAC 鉴权** | 中间件统一拦截 | 路由级 + 数据级双层校验 |
| **API 防刷** | 速率限制（Rate Limiting） | 每用户每分钟 100 次 |
| **密码** | bcrypt 哈希 | 12 轮 salt |
| **SSO** | OAuth 2.0 / OIDC（预留） | 企业微信 / 飞书 SSO |

### 13.2 数据安全

| 措施 | 说明 |
|------|------|
| **数据库加密** | 敏感字段（密码等）加密存储 |
| **传输加密** | 所有 API 走 HTTPS |
| **文件加密** | NAS 文件不加密（性能考量），云存储使用服务端加密 |
| **备份加密** | 备份文件 AES-256 加密 |
| **数据隔离** | 项目间数据严格隔离，跨项目查询需显式授权 |

### 13.3 文件安全

| 措施 | 说明 |
|------|------|
| **上传文件类型校验** | MIME 类型 + 魔数双重校验 |
| **上传文件大小限制** | 单文件 100GB，每日总量 500GB |
| **病毒扫描** | ClamAV 定时扫描（可选） |
| **路径遍历防护** | 文件名过滤 `../`、`..\\` 等 |
| **分享链接安全** | 随机 32 位 token，不可猜测 |
